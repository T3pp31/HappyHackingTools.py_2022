# スキルクリエイター

新しいスキルを作成し、反復的に改善するためのスキルです。

大まかな流れは以下の通りです：

- スキルに何をさせたいか、どのように動作させたいかを決める
- スキルのドラフトを書く
- いくつかのテストプロンプトを作成し、スキルを使ったClaudeでそれらを実行する
- ユーザーが定性的・定量的に結果を評価するのを手伝う
  - 実行がバックグラウンドで進んでいる間に、定量的な評価がまだなければドラフトを作成する（既にあれば、そのまま使うか変更が必要そうなら修正する）。そしてそれをユーザーに説明する
  - `eval-viewer/generate_review.py`スクリプトを使って、ユーザーに結果を見せ、定量的指標も確認してもらう
- ユーザーの評価からのフィードバック（および定量ベンチマークで明らかになった問題点）に基づいてスキルを書き直す
- 満足するまで繰り返す
- テストセットを拡張し、より大きなスケールで再試行する

このスキルを使う際のあなたの役割は、ユーザーがこのプロセスのどの段階にいるかを把握し、次のステージに進む手助けをすることです。例えば、ユーザーが「Xのためのスキルを作りたい」と言ったら、具体的な内容を絞り込み、ドラフトを書き、テストケースを作成し、評価方法を決め、全てのプロンプトを実行し、繰り返す手助けをします。

一方で、ユーザーが既にスキルのドラフトを持っている場合は、直接評価/反復のフェーズに進めます。

もちろん、柔軟に対応すべきで、ユーザーが「いちいち評価を回さなくていいから、一緒に感覚で作っていこう」と言えば、それに従います。

スキルが完成した後（順序は柔軟に対応して構いません）、専用のスクリプトでスキルの説明文を最適化することもできます。

いいですか？では始めましょう。

## ユーザーとのコミュニケーション

スキルクリエイターは、コーディング用語への慣れ度合いが大きく異なるユーザーに使われる可能性があります。Claudeの力によって、配管工がターミナルを開いたり、親や祖父母が「npmのインストール方法」を検索したりするトレンドが始まっています。一方で、大多数のユーザーはそれなりにコンピュータに詳しいでしょう。

ユーザーの文脈から手がかりを読み取って、コミュニケーションの表現を調整してください！デフォルトの目安として：

- 「評価」「ベンチマーク」は境界線上だが問題なし
- 「JSON」「アサーション」については、ユーザーがそれらを理解していると確信できる手がかりがない限り、説明なしに使わない

迷ったら簡単に用語を説明して構いません。ユーザーが理解できるか不確かな場合は、短い定義を添えて明確にしてください。

## スキルの作成

### 意図の把握

まず、ユーザーの意図を理解することから始めます。現在の会話に、ユーザーがキャプチャしたいワークフローが既に含まれているかもしれません（例：「これをスキルにして」）。その場合、まず会話履歴から答えを抽出します — 使用されたツール、手順の順序、ユーザーが行った修正、観察された入出力形式。ユーザーに不足部分を補ってもらい、次のステップに進む前に確認してもらいます。

1. このスキルでClaudeに何をさせたいか？
2. いつこのスキルがトリガーされるべきか？（どんなユーザーのフレーズ/コンテキストで）
3. 期待される出力形式は？
4. スキルが動作するか確認するためのテストケースを設定すべきか？客観的に検証可能な出力を持つスキル（ファイル変換、データ抽出、コード生成、固定ワークフローステップ）はテストケースの恩恵を受けます。主観的な出力を持つスキル（文体、アート）は通常不要です。スキルの種類に基づいて適切なデフォルトを提案しつつ、ユーザーに判断を委ねます。

### インタビューとリサーチ

エッジケース、入出力形式、サンプルファイル、成功基準、依存関係について積極的に質問します。この部分が固まるまでテストプロンプトの作成は待ちます。

利用可能なMCPを確認 — リサーチに有用な場合（ドキュメントの検索、類似スキルの発見、ベストプラクティスの調査）、サブエージェントが利用可能ならば並列でリサーチし、そうでなければインラインで行います。ユーザーの負担を減らすためにコンテキストを準備しておきます。

### SKILL.mdの作成

ユーザーインタビューに基づいて、以下のコンポーネントを記入します：

- **name**: スキル識別子
- **description**: いつトリガーするか、何をするか。これが主要なトリガーメカニズムです — スキルが何をするかと、いつ使うべきかの具体的なコンテキストの両方を含めます。「いつ使うか」の情報は全てここに入れ、本文には入れません。注意：現在Claudeにはスキルを「トリガーし損なう」傾向があります — 有用な場面で使わないことがあります。これに対抗するため、スキルの説明文を少し「積極的」にしてください。例えば、「内部データを表示するシンプルで高速なダッシュボードの構築方法」ではなく、「内部データを表示するシンプルで高速なダッシュボードの構築方法。ユーザーがダッシュボード、データ可視化、内部指標について言及したり、明示的に『ダッシュボード』と言わなくても何らかの会社データを表示したい場合は必ずこのスキルを使用してください。」のようにします。
- **compatibility**: 必要なツール、依存関係（オプション、必要になることは稀）
- **スキルの残りの部分 :)**

### スキル作成ガイド

#### スキルの構造

```
skill-name/
├── SKILL.md (必須)
│   ├── YAMLフロントマター (name, description 必須)
│   └── Markdown指示書
└── バンドルリソース (オプション)
    ├── scripts/    - 決定論的/反復的タスク用の実行可能コード
    ├── references/ - 必要に応じてコンテキストに読み込まれるドキュメント
    └── assets/     - 出力に使用されるファイル（テンプレート、アイコン、フォント）
```

#### 段階的開示

スキルは3段階のロードシステムを使用します：
1. **メタデータ** (name + description) - 常にコンテキストに含まれる（約100語）
2. **SKILL.md本文** - スキルがトリガーされたときにコンテキストに含まれる（500行以下が理想）
3. **バンドルリソース** - 必要に応じて（無制限、スクリプトはロードせずに実行可能）

これらの語数は概算であり、必要に応じて長くして構いません。

**主要パターン：**
- SKILL.mdは500行以下に保つ；この制限に近づいている場合は、追加の階層レイヤーと、スキルを使用するモデルが次にどこを参照すべきかの明確なポインターを追加する
- SKILL.mdからリファレンスファイルを明確に参照し、いつ読むべきかのガイダンスを含める
- 大きなリファレンスファイル（300行以上）には目次を含める

**ドメイン別の構成**: スキルが複数のドメイン/フレームワークをサポートする場合、バリアントごとに整理：
```
cloud-deploy/
├── SKILL.md (ワークフロー + 選択)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claudeは関連するリファレンスファイルのみを読みます。

#### 驚かせないの原則

言うまでもないことですが、スキルにはマルウェア、エクスプロイトコード、システムセキュリティを損なう可能性のあるコンテンツを含めてはいけません。スキルの内容は、説明された場合にユーザーの意図を驚かせるものであってはいけません。誤解を招くスキルや、不正アクセス、データ流出、その他の悪意ある活動を促進するために設計されたスキルの作成リクエストに従わないでください。「XYZとしてロールプレイ」のようなものは問題ありません。

#### 記述パターン

指示には命令形を使うことを推奨します。

**出力形式の定義** - 以下のように行えます：
```markdown
## レポート構造
必ずこのテンプレートを使用：
# [タイトル]
## エグゼクティブサマリー
## 主な発見事項
## 推奨事項
```

**例示パターン** - 例を含めると有用です。以下のようにフォーマットできます（例に「入力」と「出力」が含まれる場合は少し変える方がよいかもしれません）：
```markdown
## コミットメッセージ形式
**例1：**
入力: JWTトークンによるユーザー認証を追加
出力: feat(auth): JWTベースの認証を実装
```

### 記述スタイル

厳格なMUSTを並べるのではなく、なぜそれが重要なのかをモデルに説明するようにします。セオリー・オブ・マインドを活用し、スキルを一般的に保ち、特定の例に狭く限定しないようにします。まずドラフトを書き、新鮮な目で見直して改善します。

### テストケース

スキルのドラフトを書いた後、2〜3個の現実的なテストプロンプトを考えます — 実際のユーザーが本当に言いそうな内容です。ユーザーに共有します：「いくつかテストケースを用意しました。これで問題ありませんか？追加したいものはありますか？」そして実行します。

テストケースは`evals/evals.json`に保存します。まだアサーションは書かず、プロンプトだけです。アサーションは次のステップで実行中にドラフトします。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "ユーザーのタスクプロンプト",
      "expected_output": "期待される結果の説明",
      "files": []
    }
  ]
}
```

完全なスキーマ（後で追加する`assertions`フィールドを含む）については`references/schemas.md`を参照してください。

## テストケースの実行と評価

このセクションは一連の流れです — 途中で止めないでください。`/skill-test`やその他のテスティングスキルは使用しないでください。

結果は`<skill-name>-workspace/`に、スキルディレクトリの兄弟として配置します。ワークスペース内では、イテレーションごとに結果を整理し（`iteration-1/`、`iteration-2/`等）、各テストケースにディレクトリを割り当てます（`eval-0/`、`eval-1/`等）。これらを全て事前に作成せず、進めながら作成します。

### ステップ1：全ての実行（スキルあり・ベースライン）を同じターンで起動する

各テストケースについて、同じターンで2つのサブエージェントを起動します — 1つはスキルあり、1つはスキルなし。これは重要です：スキルありの実行を先に起動してから後でベースラインを実行するのではなく、全てを一度に起動して同じ頃に完了するようにします。

**スキルあり実行：**

```
以下のタスクを実行してください：
- スキルパス: <path-to-skill>
- タスク: <eval prompt>
- 入力ファイル: <eval files があれば、なければ "none">
- 出力保存先: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 保存する出力: <ユーザーが気にするもの — 例: ".docxファイル", "最終CSV">
```

**ベースライン実行**（同じプロンプト、ベースラインはコンテキストにより異なる）：
- **新しいスキルを作成する場合**: スキルなし。同じプロンプト、スキルパスなし、`without_skill/outputs/`に保存。
- **既存のスキルを改善する場合**: 旧バージョン。編集前にスキルをスナップショット（`cp -r <skill-path> <workspace>/skill-snapshot/`）し、ベースラインのサブエージェントをスナップショットに向けます。`old_skill/outputs/`に保存。

各テストケースに`eval_metadata.json`を書きます（アサーションは今は空で構いません）。各evalにテスト内容を説明する名前を付けます — 単なる"eval-0"ではなく。この名前をディレクトリ名にも使います。このイテレーションで新規または修正されたevalプロンプトを使う場合、各新しいevalディレクトリにこれらのファイルを作成します — 前のイテレーションから引き継がれると仮定しないでください。

```json
{
  "eval_id": 0,
  "eval_name": "わかりやすい名前をここに",
  "prompt": "ユーザーのタスクプロンプト",
  "assertions": []
}
```

### ステップ2：実行中にアサーションをドラフトする

実行が完了するのを待つだけでなく、この時間を生産的に使います。各テストケースの定量的アサーションをドラフトし、ユーザーに説明します。アサーションが既に`evals/evals.json`にある場合は、それらをレビューして何をチェックするか説明します。

良いアサーションは客観的に検証可能で、わかりやすい名前が付いています — ベンチマークビューアーで一目見ただけで、各アサーションが何をチェックするか即座に理解できるべきです。主観的なスキル（文体、デザイン品質）は定性的に評価する方が良く、人間の判断が必要なものにアサーションを無理に当てはめないでください。

アサーションをドラフトしたら`eval_metadata.json`ファイルと`evals/evals.json`を更新します。また、ビューアーで何が見えるか — 定性的な出力と定量的ベンチマークの両方 — をユーザーに説明します。

### ステップ3：実行完了時にタイミングデータを記録する

各サブエージェントタスクが完了すると、`total_tokens`と`duration_ms`を含む通知を受け取ります。このデータを直ちに実行ディレクトリ内の`timing.json`に保存します：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

このデータをキャプチャできるのはこの機会だけです — タスク通知を通じて届き、他のどこにも永続化されません。バッチ処理しようとせず、各通知が届いたら即座に処理します。

### ステップ4：採点、集計、ビューアーの起動

全ての実行が完了したら：

1. **各実行を採点する** — 採点サブエージェントを起動（またはインラインで採点）し、`agents/grader.md`を読んで各アサーションを出力に対して評価します。結果を各実行ディレクトリの`grading.json`に保存します。grading.jsonのexpectations配列は`text`、`passed`、`evidence`フィールドを使用する必要があります（`name`/`met`/`details`やその他のバリエーションではなく）— ビューアーはこれらの正確なフィールド名に依存しています。プログラムで確認できるアサーションについては、目視ではなくスクリプトを書いて実行します — スクリプトの方が高速で信頼性が高く、イテレーション間で再利用できます。

2. **ベンチマークに集計する** — skill-creatorディレクトリから集計スクリプトを実行します：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   これにより、各設定のpass_rate、時間、トークンを含む`benchmark.json`と`benchmark.md`が、平均±標準偏差と差分と共に生成されます。benchmark.jsonを手動で生成する場合は、ビューアーが期待する正確なスキーマについて`references/schemas.md`を参照してください。
各with_skillバージョンを対応するベースラインの前に配置します。

3. **アナリストパスを実行する** — ベンチマークデータを読んで、集計統計が隠してしまうパターンを表面化させます。何を探すかについては`agents/analyzer.md`（「ベンチマーク結果の分析」セクション）を参照してください — スキルに関係なく常に合格するアサーション（非弁別的）、分散の高いeval（不安定な可能性）、時間/トークンのトレードオフなど。

4. **ビューアーを起動する** — 定性的な出力と定量的データの両方を含めて：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   イテレーション2以降では、`--previous-workspace <workspace>/iteration-<N-1>`も渡します。

   **Cowork / ヘッドレス環境:** `webbrowser.open()`が利用できない、またはディスプレイがない環境では、サーバーを起動する代わりに`--static <output_path>`を使ってスタンドアロンHTMLファイルを書き出します。フィードバックは、ユーザーが「全レビューを送信」をクリックすると`feedback.json`ファイルとしてダウンロードされます。ダウンロード後、次のイテレーションで取得できるようにワークスペースディレクトリに`feedback.json`をコピーします。

注意：ビューアーの作成にはgenerate_review.pyを使用してください；カスタムHTMLを書く必要はありません。

5. **ユーザーに伝える** — 例えば：「ブラウザに結果を開きました。2つのタブがあります — 『出力結果』では各テストケースをクリックしてフィードバックを残せます。『ベンチマーク』では定量比較が見られます。確認が終わったら戻ってきて教えてください。」

### ユーザーがビューアーで見る内容

「出力結果」タブは一度に1つのテストケースを表示します：
- **プロンプト**: 与えられたタスク
- **出力**: スキルが生成したファイル、可能な場合はインライン表示
- **前回の出力**（イテレーション2以降）: 前回のイテレーションの出力を表示する折りたたみセクション
- **定量評価**（採点が実行された場合）: アサーションの合格/不合格を表示する折りたたみセクション
- **フィードバック**: 入力中に自動保存されるテキストボックス
- **前回のフィードバック**（イテレーション2以降）: 前回のコメント、テキストボックスの下に表示

「ベンチマーク」タブには統計サマリーが表示されます：各設定の合格率、タイミング、トークン使用量、evalごとの内訳とアナリストの所見。

ナビゲーションは前へ/次へボタンまたは矢印キーで行います。完了したら「全レビューを送信」をクリックし、全フィードバックが`feedback.json`に保存されます。

### ステップ5：フィードバックを読む

ユーザーが完了を告げたら、`feedback.json`を読みます：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "チャートに軸ラベルがない", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完璧、これいいね", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空のフィードバックはユーザーが問題ないと思ったことを意味します。具体的な不満があったテストケースに改善を集中させます。

使い終わったらビューアーサーバーを停止します：

```bash
kill $VIEWER_PID 2>/dev/null
```

## スキルの改善

これがループの核心です。テストケースを実行し、ユーザーが結果をレビューし、そのフィードバックに基づいてスキルをより良くする必要があります。

### 改善の考え方

1. **フィードバックから一般化する。** 大局的に見ると、ここで行っているのは、多くの異なるプロンプトに対して何百万回も（文字通り、あるいはそれ以上かもしれません）使えるスキルを作ることです。ここではあなたとユーザーは、素早く進めるためにほんの数例で反復しています。ユーザーはこれらの例を熟知しており、新しい出力を素早く評価できます。しかし、共同開発しているスキルがそれらの例に対してのみ機能するなら、使い物になりません。細かすぎるオーバーフィッティングの変更や、過度に制約的なMUSTを追加するのではなく、頑固な問題がある場合は、異なるメタファーを使ったり、異なるワーキングパターンを推奨したりしてみてください。試すコストは比較的低く、素晴らしい結果が得られるかもしれません。

2. **プロンプトを簡潔に保つ。** 効果を発揮していないものを削除します。最終的な出力だけでなくトランスクリプトも必ず読みます — スキルがモデルに生産的でないことに時間を浪費させているようであれば、それを引き起こしているスキルの部分を削除して結果を見てみます。

3. **理由を説明する。** モデルに求めていることの全ての**理由**を説明する努力をします。今日のLLMは*賢い*です。良いセオリー・オブ・マインドを持ち、良いハーネスが与えられれば、単なる指示の実行を超えて本当に成果を出せます。ユーザーからのフィードバックが簡潔だったりイライラしたものだったりしても、タスクを本当に理解し、ユーザーがなぜそう書いたのか、実際に何を書いたのかを理解し、その理解を指示に反映させてください。ALWAYS（常に）やNEVER（決して）を大文字で書いたり、超硬直的な構造を使ったりしていたら、それは警告信号です — 可能であれば、フレームを変えて推論を説明し、要求していることがなぜ重要なのかをモデルが理解できるようにします。それがより人間的で、強力で、効果的なアプローチです。

4. **テストケース間の重複作業を探す。** テスト実行のトランスクリプトを読んで、サブエージェントが全て独立して同様のヘルパースクリプトを書いたり、何かに対して同じマルチステップアプローチを取ったりしていないか注意します。3つのテストケース全てがサブエージェントに`create_docx.py`や`build_chart.py`を書かせた場合、スキルがそのスクリプトをバンドルすべきという強いシグナルです。一度書いて`scripts/`に入れ、スキルにそれを使うよう指示します。これにより、将来の全ての呼び出しが車輪の再発明をする必要がなくなります。

このタスクは非常に重要です（ここでは年間数十億の経済的価値を生み出そうとしています！）。思考時間がボトルネックではありません。じっくり時間をかけてよく考えてください。ドラフト改訂を書いてから、新鮮な目で見直して改善することをお勧めします。本当にユーザーの立場に立って、ユーザーが何を望み、何を必要としているかを理解する最善の努力をしてください。

### イテレーションループ

スキルを改善した後：

1. 改善をスキルに適用する
2. 全てのテストケースを新しい`iteration-<N+1>/`ディレクトリに再実行する（ベースライン実行も含む）。新しいスキルを作成する場合、ベースラインは常に`without_skill`（スキルなし）です — これはイテレーション間で変わりません。既存のスキルを改善する場合は、ベースラインとして何が適切か判断してください：ユーザーが持ってきたオリジナルバージョンか、前のイテレーションか。
3. `--previous-workspace`で前のイテレーションを指定してレビューアーを起動する
4. ユーザーがレビューして完了を告げるのを待つ
5. 新しいフィードバックを読み、再度改善し、繰り返す

以下の条件で終了します：
- ユーザーが満足と言った
- フィードバックが全て空（全て問題なし）
- 意味のある進歩が見られない

## 上級編：ブラインド比較

2つのスキルバージョンのより厳密な比較が必要な状況（例：ユーザーが「新バージョンは本当に良くなった？」と聞く場合）のために、ブラインド比較システムがあります。詳細は`agents/comparator.md`と`agents/analyzer.md`を参照してください。基本的な考え方は：2つの出力をどちらがどちらか知らせずに独立したエージェントに渡し、品質を判断させます。そして勝者が勝った理由を分析します。

これはオプションで、サブエージェントが必要で、ほとんどのユーザーには不要です。人間によるレビューループで通常は十分です。

## 説明文の最適化

SKILL.mdフロントマターのdescriptionフィールドは、Claudeがスキルを呼び出すかどうかを決定する主要なメカニズムです。スキルの作成または改善後、トリガー精度を向上させるために説明文の最適化を提案します。

### ステップ1：トリガー評価クエリの生成

20個の評価クエリを作成します — トリガーすべきものとすべきでないものの組み合わせ。JSONとして保存：

```json
[
  {"query": "ユーザーのプロンプト", "should_trigger": true},
  {"query": "別のプロンプト", "should_trigger": false}
]
```

クエリは現実的で、Claude CodeまたはClaude.aiのユーザーが実際に入力するようなものでなければなりません。抽象的なリクエストではなく、具体的で詳細なリクエストです。例えば、ファイルパス、ユーザーの仕事や状況に関する個人的なコンテキスト、カラム名と値、会社名、URL。少しのバックストーリー。小文字や略語、タイプミス、カジュアルな表現を含むものもあります。異なる長さのものを混ぜ、明確なものよりもエッジケースに焦点を当てます（ユーザーが確認する機会があります）。

悪い例: `"このデータをフォーマットして"`, `"PDFからテキストを抽出"`, `"チャートを作って"`

良い例: `"上司からxlsxファイルが送られてきたんだけど（ダウンロードフォルダにあって、'Q4売上最終版_v2.xlsx'みたいな名前だったと思う）、利益率をパーセンテージで表示する列を追加してほしいって言われてるんだ。売上がC列でコストがD列だと思う"`

**トリガーすべきクエリ**（8〜10個）については、カバレッジを考えます。同じ意図の異なる表現 — フォーマルなものとカジュアルなもの — が必要です。ユーザーがスキルやファイルタイプを明示的に名前を挙げていないが明らかに必要としているケースを含めます。珍しいユースケースや、このスキルが他のスキルと競合するが勝つべきケースも含めます。

**トリガーすべきでないクエリ**（8〜10個）については、最も価値があるのはニアミス — スキルとキーワードや概念を共有しているが、実際には異なるものが必要なクエリです。隣接ドメイン、単純なキーワードマッチならトリガーするが本来すべきでない曖昧な表現、スキルが扱うことに触れているが別のツールがより適切なコンテキストのケースを考えます。

避けるべき重要な点：トリガーすべきでないクエリを明らかに無関係にしないでください。PDFスキルの否定テストとしての「フィボナッチ関数を書いて」は簡単すぎて、何もテストしません。否定ケースは本当にトリッキーであるべきです。

### ステップ2：ユーザーとレビュー

HTMLテンプレートを使って評価セットをユーザーにレビュー用に提示します：

1. `assets/eval_review.html`からテンプレートを読む
2. プレースホルダーを置換：
   - `__EVAL_DATA_PLACEHOLDER__` → eval項目のJSON配列（引用符なし — JS変数の代入です）
   - `__SKILL_NAME_PLACEHOLDER__` → スキル名
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → スキルの現在の説明文
3. 一時ファイルに書き出し（例：`/tmp/eval_review_<skill-name>.html`）、開く：`open /tmp/eval_review_<skill-name>.html`
4. ユーザーはクエリの編集、should-triggerの切り替え、エントリの追加/削除が可能で、「評価セットをエクスポート」をクリック
5. ファイルは`~/Downloads/eval_set.json`にダウンロードされます — 複数ある場合（例：`eval_set (1).json`）に備えてダウンロードフォルダで最新版を確認

このステップは重要です — 悪い評価クエリは悪い説明文につながります。

### ステップ3：最適化ループの実行

ユーザーに伝えます：「これには時間がかかります — バックグラウンドで最適化ループを実行し、定期的に確認します。」

評価セットをワークスペースに保存し、バックグラウンドで実行します：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

システムプロンプトのモデルID（現在のセッションを動かしているもの）を使用し、トリガーテストがユーザーの実際の体験と一致するようにします。

実行中、定期的に出力をtailして、現在のイテレーション番号とスコアの状況をユーザーに更新します。

これは完全な最適化ループを自動的に処理します。評価セットを60%の訓練と40%のホールドアウトテストに分割し、現在の説明文を評価（信頼性のあるトリガー率を得るために各クエリを3回実行）し、失敗に基づいて改善を提案するために拡張思考付きのClaudeを呼び出します。訓練とテストの両方で各新しい説明文を再評価し、最大5回反復します。完了すると、イテレーションごとの結果を表示するHTMLレポートをブラウザで開き、`best_description`を含むJSONを返します — オーバーフィッティングを避けるため、訓練スコアではなくテストスコアで選択されます。

### スキルのトリガーの仕組み

トリガーメカニズムを理解することで、より良い評価クエリの設計に役立ちます。スキルはClaudeの`available_skills`リストにname + descriptionで表示され、Claudeはその説明文に基づいてスキルを参照するかどうかを決定します。重要なのは、Claudeは自分で簡単に処理できるタスクにはスキルを参照しないということです — 「このPDFを読んで」のような単純な1ステップのクエリは、説明文が完璧に一致していても、Claudeが基本ツールで直接処理できるため、スキルをトリガーしない場合があります。複雑な、マルチステップの、または専門的なクエリは、説明文が一致していれば確実にスキルをトリガーします。

これは、評価クエリがClaude がスキルを参照することで実際に恩恵を受けるほど十分に実質的である必要があることを意味します。「ファイルXを読んで」のような単純なクエリは良いテストケースではありません — 説明文の品質に関係なくスキルをトリガーしません。

### ステップ4：結果の適用

JSON出力から`best_description`を取得し、スキルのSKILL.mdフロントマターを更新します。ユーザーにビフォー/アフターを見せ、スコアを報告します。

### パッケージングと提示（`present_files`ツールが利用可能な場合のみ）

`present_files`ツールにアクセスできるか確認します。できない場合はこのステップをスキップします。できる場合は、スキルをパッケージングして.skillファイルをユーザーに提示します：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

パッケージング後、ユーザーにインストールできるように`.skill`ファイルのパスを案内します。

## Claude.ai固有の手順

Claude.aiでは、コアワークフローは同じ（ドラフト → テスト → レビュー → 改善 → 繰り返し）ですが、Claude.aiにはサブエージェントがないため、一部の仕組みが変わります。以下の点を適応してください：

**テストケースの実行**: サブエージェントがないため並列実行はできません。各テストケースについて、スキルのSKILL.mdを読んでから、その指示に従ってテストプロンプトを自分で実行します。1つずつ行います。これは独立したサブエージェントよりも厳密さに欠けます（あなたがスキルを書き、それを実行するので、フルコンテキストを持っています）が、有用なサニティチェックです — 人間のレビューステップがそれを補います。ベースライン実行はスキップし、要求されたタスクをスキルを使って完了させます。

**結果のレビュー**: ブラウザを開けない場合（例：Claude.aiのVMにディスプレイがない、リモートサーバーにいる）、ブラウザレビューアーは完全にスキップします。代わりに、会話内で直接結果を提示します。各テストケースについて、プロンプトと出力を表示します。ユーザーが見る必要のある出力がファイル（.docxや.xlsxなど）の場合、ファイルシステムに保存してユーザーにダウンロードして確認できるよう場所を伝えます。インラインでフィードバックを求めます：「これはどう見えますか？何か変更したいことはありますか？」

**ベンチマーク**: 定量的ベンチマークはスキップします — サブエージェントなしではベースライン比較が意味を持ちません。ユーザーからの定性的フィードバックに集中します。

**イテレーションループ**: 以前と同様 — スキルを改善し、テストケースを再実行し、フィードバックを求めます — ただし途中のブラウザレビューアーはありません。ファイルシステムがあれば、イテレーションディレクトリに結果を整理することは引き続き可能です。

**説明文の最適化**: このセクションには`claude` CLIツール（具体的には`claude -p`）が必要で、Claude Codeでのみ利用可能です。Claude.aiでは飛ばします。

**ブラインド比較**: サブエージェントが必要です。スキップします。

**パッケージング**: `package_skill.py`スクリプトはPythonとファイルシステムがあればどこでも動きます。Claude.aiでは実行でき、ユーザーは結果の`.skill`ファイルをダウンロードできます。

## Cowork固有の手順

Coworkの場合、主に知っておくべきことは：

- サブエージェントが利用可能なので、メインワークフロー（テストケースの並列起動、ベースライン実行、採点など）は全て機能します。（ただし、タイムアウトで深刻な問題が発生した場合は、並列ではなく直列でテストプロンプトを実行しても構いません。）
- ブラウザやディスプレイがないため、eval ビューアーを生成する際は、サーバーを起動する代わりに`--static <output_path>`を使ってスタンドアロンHTMLファイルを書き出します。そしてユーザーがクリックしてブラウザでHTMLを開けるリンクを提供します。
- 何らかの理由で、Cowork環境ではClaudeがテスト実行後にevalビューアーを生成しない傾向があるため、改めて強調します：CoworkでもClaude Codeでも、テスト実行後は必ずevalビューアーを生成して、あなた自身がスキルを見直して修正する前に人間に例を見てもらいます。`generate_review.py`を使用してください（独自のHTMLコードを書くのではなく）。大文字で強調させてください：あなた自身が入力を評価する**前に**evalビューアーを生成してください。人間にできるだけ早く見てもらいたいのです！
- フィードバックの仕組みが異なります：実行中のサーバーがないため、ビューアーの「全レビューを送信」ボタンは`feedback.json`をファイルとしてダウンロードします。そこから読み込めます（最初にアクセスをリクエストする必要があるかもしれません）。
- パッケージングは動作します — `package_skill.py`はPythonとファイルシステムだけで十分です。
- 説明文の最適化（`run_loop.py` / `run_eval.py`）は、サブプロセス経由で`claude -p`を使うためブラウザではなく、Coworkでも正常に動作するはずです。ただし、スキルが完全に完成し、ユーザーが良い状態だと同意するまで保存しておいてください。
